# Спецификация формата архивов .pkg (Glyphix OS)
**Статус:** Продвинутый реверс-инжиниринг (Включает анализ байткода ВМ и графики)
**Целевая аппаратная платформа:** Actions Semiconductor ATS3085S (Смарт-часы DT G1)
**ОС:** Glyphix Framework
**Порядок байтов (Endianness):** Little-Endian
**Выравнивание данных (Alignment):** 16 байт (`0x10`)

---

## 1. Общая топология архива
Файл `.pkg` является линейным потоковым некомпрессированным архивом.
Физическая структура:
1. Глобальный заголовок (32 байта).
2. Таблица смещений файлов (размер: `Количество файлов * 16 байт`).
3. Таблица имен файлов (ASCII-строки, склеенные без нуль-терминаторов).
4. Блок выравнивания нулевыми байтами до границы кратной 16 (`0x10`).
5. Блок полезной нагрузки (данные каждого файла всегда начинаются с адреса, кратного 16).

---

## 2. Глобальный заголовок (Global Header)
Располагается строго по смещению `0x0000`. Выровнен до 32 байт.

```c
struct pkg_header {
    uint8_t  magic[2];    // 0x00: Строгая сигнатура 0x43, 0x52 ("CR")
    uint16_t unknown;     // 0x02: Назначение неизвестно (стандартно 0x0000)
    uint32_t file_count;  // 0x04: Количество файлов в архиве
    uint32_t reserved1;   // 0x08: Зарезервировано (0x00000000)
    uint32_t reserved2;   // 0x0C: Зарезервировано (0x00000000)
    uint8_t  padding[16]; // 0x10: Нулевые байты выравнивания до 0x20
};
```

---

## 3. Таблица файлов (File Offset Table)
Начинается со смещения `0x0020`. Массив структур метаданных.

```c
struct pkg_file_entry {
    uint32_t name_hash;   // 0x00: 32-битный хеш пути файла (алгоритм исследуется)
    uint32_t data_offset; // 0x04: Абсолютное смещение к полезной нагрузке файла
    uint32_t data_size;   // 0x08: Размер файла в байтах
    uint32_t name_length; // 0x0C: Длина пути файла в байтах
};
```

---

## 4. Таблица путей (String Table)
Находится сразу после массива `pkg_file_entry`. 
* Представляет собой непрерывный блок текста (например, `main/index.jsmain/index.uoapp.js`).
* Парсинг осуществляется строго путем отсчета `name_length` байт для каждого файла из таблицы смещений. `\0` в качестве разделителя не используется.

---

## 5. Форматы полезной нагрузки (Payloads)

Компилятор Glyphix (`gx build`) применяет AOT-компиляцию для большинства ресурсов на этапе сборки пакета.

### 5.1. Скомпилированный JavaScript Bytecode (`*.js`)
Скрипты транслируются в бинарный байткод для встроенной виртуальной машины (JS Engine).

* **Магическая сигнатура:** `0x00 - 0x03`: `F2 DA 50 01` (Little-Endian: `0x0150DAF2`).
* **Метаданные заголовка:** `0x04 - 0x09` (6 байт) — зависят от контрольной суммы или длины пула констант.
* **Пул констант (Constant Pool / String Pool):**
  Начинается со смещения `0x000A`. Хранит все используемые в коде строки: полные URI пакета, вызовы `console`, имена методов жизненного цикла (`onCreate`, `onShow`) и системные импорты.
  **Алгоритм упаковки длин (Bitwise Length Encoding):**
  Длина строки зашифрована в байте-префиксе. Виртуальная машина использует побитовый сдвиг вправо:
  `String_Length = Prefix_Byte >> 1` (или `Prefix_Byte / 2`).
  *Пример:* Байт `0x48` (`72` dec) означает длину строки в 36 символов.
* **Секция опкодов (VM Opcodes):**
  Начало блока бинарных процессорных инструкций виртуальной машины строго детерминировано маркером входа (Entry Point Marker): `0F 9E 03`.

### 5.2. UI-байткод (Разметка и стили `*.uo`)
Конвертированный бинарный формат исходников `*.ux`. Отвечает за DOM и CSS.
* **Прекомпиляция CSS:** Стили не хранятся как текст. Движок парсит CSS на этапе сборки и инжектирует готовые 32-битные токены (например, RGBA `33 33 33 FF` для цвета `#333333`).
* **DOM (Структура элементов):** Дерево тегов (`Div`, `Image`, `Text`) и их атрибуты (`class`, `src`) сериализуются в формате TLV (Type-Length-Value). Имена тегов сохраняются текстом.
* **Биндинг событий:** Директивы вида `on:click="exitApp"` аппаратно транслируются в явные вызовы методов контекста компонента (префикс `this.`, например, строка `this.exitApp`).

### 5.3. Формат изображений PGF (Glyphix Picture Format)
Бинарный формат квантованных изображений (оптимизирован для DMA2D акселератора MCU).
Общий размер заголовка: 40 байт (`0x28`).

```c
struct pgf_header {
    uint8_t  magic[4];       // 0x00: 0x50, 0x47, 0x46, 0x00 ("PGF\0")
    uint8_t  reserved1[24];  // 0x04: Padding 
    uint32_t unknown_header; // 0x18: Неизвестный флаг (обычно 0x0000001C)
    uint16_t width;          // 0x1C: Ширина (в пикселях)
    uint16_t height;         // 0x1E: Высота (в пикселях)
    uint16_t pitch;          // 0x20: Шаг строки / Stride (в байтах)
    uint16_t format;         // 0x22: Формат пикселей (0x0006 = 8-bit Indexed Paletted)
    uint16_t palette_colors; // 0x24: Размер палитры (0x0100 = 256 цветов)
    uint32_t payload_size;   // 0x26: Размер тела данных (Матрица + Палитра)
    // 0x2A ... Начало полезной нагрузки
};
```
* **Физическая структура данных для 8-bit Indexed:**
  1. Матрица индексов пикселей (размер `width * height` байт).
  2. Палитра цветов (Lookup Table) в формате 32-bit ARGB/BGRA (размер `palette_colors * 4` байт).
  3. Математическая проверка `payload_size`: `(width * height) + (palette_colors * 4)`.

### 5.4. Сериализованный манифест (`manifest.json`)
Хранится в частично скомпилированном бинарном формате. Строковые значения (bundle ID, название, версии) хранятся открытым текстом. Каждой строке предшествует 16-битное (Little-Endian) значение ее длины в байтах. Ключи JSON-структуры заменяются бинарными токенами.
